Now a day, user don't like waiting. Lead by Google, Facebook and Twitter, all web site must be extremely fast and comparable to native application. The trick is using javascript.

Offline Hijax

In 2004, Google outline crawling strategy for making AJAX site URL crawlable. At the begining, web master don't buy it. But now Bing also follow Google's AJAX crawling strategy, and most arguments against it are irrelevant whether semantically correct or not. 

To better understand, what we are doing, let us start with the problem in client side. 

We want offline web site, this is archieved using a new standard called Offline Application Caching APIs. It has its won limitation and quark too. Anyways the gist of it is you define list of URLs that you want the browser to cache in a manifest file, site.appcache. For example: 

CACHE MANIFEST
index.html
help.html
style/default.css
images/logo.png
images/backgound.png

Each web page link to it by declaring as:

<!DOCTYPE HTML>
<html manifest="site.appcache">

Please be note query parameters are part of URL and must register separately too. Hash fragment is not part of URL. It is not even sent to the server.

It is fine for web app (actually that is their target audience), but for web site it is difficult. The problem not end here because, if you update the app cache minifest to update the site, the browser will download all the files, of which user not likely to visit at all.

What we provide with YDN-hijax, is you attach the hijax javascript module in each of your web page and make app cache file with some base URL. Once a user visit the site, our module will lazily store into the client database and update the links into ajax url. If user click the link, a hijacked callback update the page pulling out the content from the database and redirect is cancelled.
